## 2026-06-10

### 削除コマンド（rm / del）は環境レベルでブロックされる
- ユーザーが質問で「削除OK」と答えても、`Bash` の `rm` コマンドは
  パーミッションシステムによって拒否されることがある。
- **対応**: 削除が必要な場合はユーザーにエクスプローラで手動削除を依頼する。
  Bash の rm 再試行ではなく、最初から手動削除を提案するのが効率的。

### ファイル新規作成は問題なく通る
- `Write` ツールで新しい場所にファイルを作るのは制限なく実行できる。
- 配置整理は「新場所にコピー → 旧ファイルは手動削除依頼」のフローが安全。

### CLAUDE.md の安全ルールに従う
- 既存ファイル上書き前の確認、削除コマンドの事前承認、
  パッケージ追加の事前説明、技術コマンドの日本語説明 — すべて遵守する。
- ユーザーは技術者ではないので、エラーや実行内容は必ず平易な言葉で要約する。

### 物体距離（Lobj）は「光束」と「焦点/焦線/数値読み出し」の両方に必ず反映する
- 起こったこと: Phase 2 で物体距離スライダを動かしても、光束は動くが
  前焦線・後焦線・最小錯乱円・パラメータ読み出しが固定値のまま。
- 原因: `sturmConoid` 内部で焦線位置を `backFocalDistanceFromCornea`
  （Lobj=0 専用 = 無限遠平行光の後側焦点）で計算していた。
  光束は `traceBundle` で Lobj を渡していたので動くが、マーカー/数値だけ無限遠固定。
- 対策: 任意 Lobj 用 `imageDistanceFromCornea(system, Lobj)` を追加し、
  焦線・焦点マーカー・数値読み出しもすべて Lobj 反映に統一。
- 教訓: 「物体距離スライダ」のような状態は、光束追跡だけでなく、
  焦点位置・読み出しなど派生量にも漏れなく反映する。可視化と数値の整合性は教材の信頼性に直結。

## 2026-06-11

### Phase 4 でも「視距離スライダ」が光学計算に反映されていなかった（同じパターン再発）
- 起こったこと: Phase 4 で視距離スライダを動かしても PSF も網膜像も全く変化しなかった。
- 原因: `distanceM` は読み出し（物理サイズ mm 換算）にしか使われておらず、
  `computePsfFromRefraction` には渡されていなかった。視力値スライダ（acuityDecimal）も、
  既定の点光源モードでは原理上 PSF に影響しないため「効かない」と見えた。
- 対策:
  - 無調節モデルとして `effectiveSphereD = sphereD + 1/distanceM` を計算し、
    PSF 計算にはこの実効値を渡す（例: −3D 近視眼が 0.33m を見ると effectiveSphereD ≈ 0 で鮮明）。
  - 読み出しに「等価球面（屈折異常）／調節要求／実効デフォーカス」の3行に分けて表示し、
    数値とビジュアルの対応を明示。
  - 視力値スライダのヒントに「点光源モードでは効果なし」と明記。
- 教訓:
  - **「スライダがある＝光学計算に反映される」と利用者は当然思う**。
    読み出しだけに使うパラメータは原則作らない（必要なら明示する）。
  - 2026-06-10 と同じ「スライダが派生量にしか効いていない」パターン。
    新しいフェーズを足すたびに、**全スライダが画像/物理計算のどこかに効くこと**を
    実装前にチェックリストとして確認する。
  - 視標タイプ（点光源 vs 視標）でスライダの有効/無効が変わる場合は、
    UI のヒントか動的な無効化で明示する。

### 二分法で「目的物理量の単調性方向」を取り違えていた（Phase 3 toric、当時は Phase α と呼称）
- 起こったこと: Phase α で IOL のスライダを動かしても 3D の前焦線・後焦線が一切動かない。
- 原因: `cornealRadiusFromRefractionMM`（屈折値 → 角膜半径の逆算）の二分法で、
  `ref < target` の場合に `lo = mid` とすべきところを `hi = mid` にしていた。
  屈折値は半径の単調増加関数（半径大 → 屈折力弱 → 値が大）。
  ref(r) = target を解くには、ref(mid) が target より小さいときに r をさらに大きく取りに行く必要がある。
  不等号が逆だったため、半径は常に端点（5.0 か 12.0 mm）に張り付き、しかも強・弱で同じ値に収束 → 乱視差が消えて焦線が動かないように見えていた。
- 対策: 不等号を反転。さらに「往復一致テスト（半径 → 屈折値 → 半径）」を追加して回帰防止。
- 教訓:
  - **二分法を書いたら必ず「往復一致テスト」を書く**。出力の整合だけ見るテストでは、
    内部の単調性方向のバグを見抜けない（power vector の数学テスト 7 件は全部通っていたが、
    バグは下流の二分法側にあった）。
  - 単調性の向きはコメントで明示する（`f(r) は r について単調増加` のように）。
    1 行のコメントがあれば、不等号レビュー時に瞬時に正誤が判断できる。
  - 「画面が動かない」系のバグは、状態が「どこで止まっているか」を疑う。
    今回は二分法が端点でクランプ → 上流の数学テストは通っているのに下流の物理量がフリーズ、
    という典型パターン。

### 3D シーンの「軸角度 → 方向ベクトル」変換で sin/cos を入れ違えた（Phase 2 / 3、当時は Phase α と呼称）
- 起こったこと: Phase 2・Phase α で前焦線と後焦線の向きが逆に見えた
  （倒乱視＝強主経線が水平 0° の場合、前焦線は垂直・後焦線は水平になるべきだが、入れ替わっていた）。
- 原因: `sturm3d.ts` で軸角 a から経線方向ベクトルを作る式が
  `(0, cos a, sin a)` になっており、a=0 のとき Y 軸（垂直）方向を指していた。
  入力規約は「軸 0° = 水平」「軸 90° = 垂直」だったため表示が 90° ズレ、
  さらに前/後焦線は経線に **直交** する向きに伸びるので、結果として「向きが逆」に見えた。
- 対策: `(0, sin a, cos a)` 系に変更し、IOL マーカーの軸方向ドットも同じ規約で統一。
  式の直前にコメントで「軸 0° = 水平、軸 90° = 垂直」と規約を明示。
- 教訓:
  - **角度 → 方向ベクトルの式には「角度ゼロでどの軸を指すか」のコメントを必ず添える**。
    sin/cos の取り違いは検算しにくいので、規約 1 行があると即発見できる。
  - 乱視の「経線方向」と「焦線方向」は **直交** する。
    向きの確認は「a=0 で steepDir 直接」だけでなく、
    「a=0 で前焦線（flat 方向）が垂直か」までセットで検証する。
  - 数値テスト（power vector・屈折値）は通っていても、3D の **空間的向き** は別レイヤーのバグになる。
    ビジュアルの自動検証が難しいぶん、医師目線の事実（「倒乱視ならこうなるはず」）をコメントに残す。

## 2026-06-12

### 3D マーカーの「見せ方」は内部慣習ではなく臨床慣習に合わせる（Phase 3、当時は Phase α と呼称）
- 起こったこと: Phase α で IOL 軸マーカー（橙ドット）を内部の `iolAxisDeg` 方向にそのまま描画していた。
  内部慣習で `iolAxisDeg` は「IOL の **強主経線軸**」（円柱量が最大の軸）。
  一方、実臨床の toric IOL の刻印マーカーは「IOL の **弱主経線**（円柱量最小の軸）」を示すのが標準で、
  外科医はそのマーカーを角膜の **強主経線** に合わせて挿入する。
  シミュレーションでは「橙ドットを角膜弱主経線（青タブ）に合わせると乱視キャンセル最大」になっていて、
  臨床の手の動き（「マーカーを赤に合わせる」）と逆の挙動になっていた。
- 対策: 描画時のみ方向ベクトルを 90° 回し、橙ドットを IOL 弱主経線方向に置く。
  内部計算（power vector 合成）の慣習は変えない。
  ボタン名も「軸を直交させる」→「IOL マーカーを角膜強主経線（赤タブ）に合わせる」に更新。
- 教訓:
  - **内部の数式慣習と、画面上のマーカーの「意味」は別レイヤーで設計する**。
    数式が正しくても、見え方が臨床と逆だと医師にとっては使いにくい教材になる。
  - 物理オブジェクト（toric IOL マーカー、角膜トポグラフィ表示など）を 3D に描くときは、
    「実物（または実臨床の表示）が何を示しているか」を一次資料で確認してから描画方向を決める。
  - ボタンや凡例の文言は「**画面で何が起きるか**」を主語にする方が直感的。
    「IOL 軸を角膜軸に直交させる」は数学的には正しいが、初学者には何の軸かが伝わらない。
    「橙ドットを赤タブに合わせる」のようにビジュアルを直接指す方が、術中の手順とも対応しやすい。

### フェーズ番号と内部ファイル名が乖離する場合は「信源」を 1 か所に集約する

- 起こったこと: Phase α として試験運用していた toric IOL を Phase 3 として正式採用した結果、
  内部ファイル名 `phaseToric.ts` / `phase3.ts` / `phase4.ts` と UI 上の Phase 番号
  (3 = toric, 4 = 媒質置換, 5 = PSF) がずれた状態になった。
  すべての import/関数名/型名 (`mountPhase3`, `Phase4Target` 等) を機械的に書き換えると
  テストや他箇所の整合性が広範に崩れるリスクが大きい。
- 対策: 信源を `src/main.ts` の `PHASES` 配列に一本化し、
  各ファイル冒頭コメントで「Phase 番号と内部ファイル名は一致しない、信源は main.ts」を明示。
  ROADMAP / README / todo / lessons でも UI 上の番号を使い、`(旧 Phase α)` 等の括弧で履歴を補足。
- 教訓:
  - **表示用の番号と内部識別子は別レイヤー**として扱ったほうが、後で順番が入れ替わったときの
    修正コストが小さい（教材の構成変更は珍しくない想定）。
  - 履歴を持つドキュメント（lessons.md など）は時系列の事実を変えず、当時の呼称に括弧で
    現在名を併記すると「過去のミス」と「現在のシステム」の橋渡しになる。
  - 内部リネームを強行する場合は事前に全 import / 型 / テストを洗い出して影響範囲を確定する。
    今回はシンプル最優先（CLAUDE.md コア原則）に従いリネームを見送った。
